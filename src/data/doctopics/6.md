# Способы передачи параметров. Передача по значению

## Теоретическая база: Связь вызывающего и вызываемого кода

Как было рассмотрено в механизме работы функций, любая функция в языках программирования функционирует подобно черному ящику. Чтобы этот черный ящик мог выполнить полезную работу, ему необходимо передать исходные данные, а затем — получить результат.

Процесс транспортировки данных из вызывающей среды (например, функции `main`) внутрь изолированной среды вызываемой функции называется **передачей параметров**.

В языке C++ предусмотрено несколько фундаментально разных механизмов передачи параметров, выбор которых критически влияет на производительность, безопасность и архитектуру приложения:

1. Передача по значению (Pass by Value).
2. Передача по ссылке (Pass by Reference).
3. Передача по указателю (Pass by Pointer).
4. Передача по константной ссылке (Pass by Const Reference).

Исторически первым и самым базовым методом, унаследованным C++ от языка C и более ранних систем, является **передача по значению**.

---

## Концепция передачи по значению (Pass by Value)

**Передача по значению** — это механизм, при котором в функцию передаются не сами оригинальные переменные, а их точные, но полностью независимые **копии**.

Архитектурный принцип работы этого механизма заключается в следующем:

1. Вызывающий код (Caller) вычисляет значение фактического аргумента.
2. Процессор выделяет новую память в стековом фрейме (Call Stack Frame) вызываемой функции под локальную переменную (формальный параметр).
3. Процессор копирует (дублирует) вычисленное значение из памяти вызывающего кода в только что выделенную область памяти вызываемой функции.

### Главное свойство передачи по значению: Полная изоляция

Поскольку функция работает исключительно с _копией_ данных, любые изменения и манипуляции, производимые внутри тела функции над формальными параметрами, **никак не отражаются** на оригинальных переменных в вызывающем коде.

Это свойство делает передачу по значению самым **безопасным** методом. Вы можете быть абсолютно уверены, что вызванная функция случайно или намеренно не испортит (не модифицирует) ваши исходные данные.

## Пример использования (Абсолютная изоляция)

Рассмотрим классическую задачу инкремента (увеличения значения). Мы попытаемся изменить оригинальную переменную через функцию, используя передачу по значению, чтобы наглядно доказать теорему об изоляции копий.

```cpp
#include <iostream>

// ПРОТОТИП: Функция декларирует передачу ПО ЗНАЧЕНИЮ.
// Об этом говорит отсутствие спецсимволов (& или *) после типа int.
void attemptToModify(int localCopy);

int main() {
    int originalNumber = 10;

    std::cout << "1. [main] Перед вызовом originalNumber = " << originalNumber << "\n";

    // ВЫЗОВ: Передаем значение переменной originalNumber
    attemptToModify(originalNumber);

    // СМОТРИМ РЕЗУЛЬТАТ: изменилось ли оригинальное значение?
    std::cout << "4. [main] После вызова originalNumber = " << originalNumber << "\n";

    return 0;
}

// ОПРЕДЕЛЕНИЕ: localCopy - это независимый клон
void attemptToModify(int localCopy) {
    std::cout << "   2. [func] Получена копия localCopy = " << localCopy << "\n";

    // Пытаемся изменить значение
    localCopy = localCopy + 50;

    std::cout << "   3. [func] Копия изменена. localCopy = " << localCopy << "\n";
    // При выходе из функции блок памяти localCopy будет безвозвратно уничтожен
}
```

**Вывод программы на консоль:**

```text
1. [main] Перед вызовом originalNumber = 10
   2. [func] Получена копия localCopy = 10
   3. [func] Копия изменена. localCopy = 60
4. [main] После вызова originalNumber = 10
```

Как мы видим, `originalNumber` остался равен 10. Функция изменила лишь свой временный клон.

## Протокол прохождения контрольного примера (Аппаратный уровень)

Рассмотрим, что физически происходит в оперативной памяти (в сегменте Стека) при выполнении вышеуказанного кода.

- **Шаг 1. Выполнение `int originalNumber = 10;` в `main`**
  - _Стек:_ Выделяется 4 байта (размер int) по физическому адресу (допустим) `0x7FFF0001`.
  - В эту ячейку записывается бинарное представление числа `10`.

- **Шаг 2. Вызов функции `attemptToModify(originalNumber);`**
  - Вектор исполнения готовится к переходу. Сначала вычисляется значение аргумента (чтение из `0x7FFF0001` -> получаем `10`).
  - _Стек:_ Создается новый стековый фрейм (Stack Frame) для функции `attemptToModify`.
  - Сдвигается указатель стека (Stack Pointer). Выделяется новые 4 байта для локальной переменной `localCopy` по новому адресу (допустим) `0x7FFF0010`.
  - **ОПЕРАЦИЯ КОПИРОВАНИЯ:** Значение `10` копируется в новую ячейку `0x7FFF0010`.

- **Шаг 3. Выполнение `localCopy = localCopy + 50;`**
  - Процессор обращается по адресу `0x7FFF0010`, берет `10`, прибавляет `50`.
  - Новое значение `60` записывается **исключительно** по адресу `0x7FFF0010`.
  - Ячейка `0x7FFF0001` (принадлежащая `main`) остается неприкосновенной.

- **Шаг 4. Возврат из функции `return;`**
  - _Стек:_ Фрейм функции `attemptToModify` уничтожается (указатель стека возвращается в исходное положение).
  - Ячейка памяти `0x7FFF0010`, хранившая измененное число `60`, считается свободной и будет перезаписана при следующем вызове любой другой функции (данные превращаются в мусор).
  - Управление возвращается в `main`, где по адресу `0x7FFF0001` по-прежнему лежит `10`.

## Проблема накладных расходов (Overhead)

Несмотря на абсолютную безопасность, передача по значению имеет критический инженерный недостаток: **накладные расходы на копирование**.

Для примитивных встроенных типов (`int`, `float`, `char` — размеры 4-8 байт) затраты на копирование ничтожны и занимают процессора буквально один-два такта (операция `MOV` в ассемблере).

Однако, в современном C++ функциям часто передаются огромные объекты (например, `std::vector` с миллионом элементов, массивы структур, огромные строки). Передача структур данных по значению означает, что компилятор сгенерирует цикл (или вызовет `memcpy`), который будет побайтово дублировать **ВЕСЬ** миллион элементов в новый пул памяти.

- _Результат:_ Падение производительности (от микросекунд до десятков миллисекунд), засорение кэша процессора и неконтролируемое (часто двукратное) разрастание потребления оперативной памяти в моменте вызова функции.
- _Правило C++:_ Передачу по значению следует применять **только** для примитивных встроенных типов (числа, символы, булевы флаги, маленькие структуры-обертки). Для любых "тяжелых" объектов (контейнеров) применяется передача по константной ссылке `const T&`.

## Резюме характеристик

| Характеристика              | Оценка / Описание                                                                  |
| :-------------------------- | :--------------------------------------------------------------------------------- |
| **Суть механизма**          | Передача в функцию дубликата (копии) реальных данных                               |
| **Синтаксис в прототипе**   | Тип без модификаторов: `void func(int x)`                                          |
| **Безопасность (Изоляция)** | Максимальная. Оригинал невозможно повредить изнутри функции                        |
| **Влияние на ОЗУ**          | Требует выделения доп. памяти под копию в стековом фрейме                          |
| **Производительность**      | Быстро для `int`/`double`. Катастрофически медленно для огромных объектов/массивов |
| **Что передается**          | L-value (переменные), R-value (литералы вроде `5`, математические выражения)       |

## Контрольные вопросы для самопроверки

- В чем заключается фундаментальная концепция механизма "передачи по значению" в контексте управления оперативной памятью?
- Как поведет себя функция, реализующая обмен значениями (swap), если попытаться передать ей две переменные по значению? Что в итоге поменяется местами?
- Опишите, какие действия вызовет процессор при попытке передать по значению объект `std::vector<int>`, содержащий 100 000 целых чисел. К чему это приведет в плане процессорного времени и расхода ОЗУ?
- Почему, несмотря на аппаратные затраты (дублирование данных), передача встроенных типов данных (таких как `int` или `char`) по значению в большинстве случаев работает быстрее, чем передача их же по ссылке или указателю? (Подсказка: накладные расходы на разименование в регистрах CPU).
- Чем формальный параметр при передаче по значению отличается от обычной локальной переменной, объявленной внутри тела функции? (Спойлер: ничем, за исключением способа инициализации).
