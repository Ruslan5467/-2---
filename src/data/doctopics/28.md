# Основные операции с односвязными линейными списками без головного элемента

## Теоретическая база: Зависимость от позиции

В массивах (`std::vector`) мы привыкли использовать индексы. В связных списках индексов как физически упорядоченных ячеек памяти нет. Для выполнения операций нам необходимо использовать **Указатели (Итераторы)**, проходя по списку от узла к узлу (link by link).

При реализации списка без фиктивного (dummy) головного элемента возникает важная архитектурная особенность: **первый узел (head) становится особым (краевым) случаем**.
Например, вставка или удаление узла _в середине_ списка требует только изменения указателя `next` предыдущего (предшествующего) узла. Однако, если мы удаляем первый элемент или вставляем новый элемент _в самое начало_, предшествующего узла просто не существует. В этом случае нам приходится переназначать сам внешний главный указатель `head`.

Поэтому все функции, потенциально способные изменить первый элемент (вставка, удаление), должны принимать указатель `head` **по ссылке** (`Node*& head`) или возвращать модифицированный указатель.

Рассмотрим три фундаментальные операции:

1. Поиск узла по заданному значению.
2. Вставка элемента сразу после заданного узла.
3. Удаление первого вхождения элемента по значению.

---

## 1. Поиск элемента (Search)

Поиск в связном списке осуществляется исключительно последовательным перебором, шагая по адресам от головы до того момента, пока не встретится искомое значение или пока список не закончится (`nullptr`). Асимптотическая сложность такого поиска — $O(N)$.

```cpp
// Возвращает указатель на найденный узел, либо nullptr, если элемент не найден
Node* findNode(Node* head, int targetValue) {
    Node* current = head; // Заводим бегунок

    while (current != nullptr) {
        if (current->data == targetValue) {
            return current; // Успех! Возвращаем адрес узла
        }
        current = current->next; // Шагаем на следующий вагон
    }

    return nullptr; // Вагон с таким грузом не найден
}
```

## 2. Вставка после заданного элемента (Insert After)

Если у нас уже есть физический адрес (указатель) какого-либо узла в списке (например, полученный из функции `findNode`), мы можем встроить за ним новый узел мгновенно, за идеальное время **$O(1)$**. Это главное преимущество списков перед массивами.

**Механика хирургической сшивки:**

1. Выделяем память под новый узел (`newNode`).
2. Привязываем "хвост" списка к новому узлу (`newNode->next = targetNode->next`).
3. Привязываем целевой узел к нашему новому узлу (`targetNode->next = newNode`).

> **Внимание:** Порядок шагов 2 и 3 является критическим. Если их поменять местами, мы навсегда потеряем адрес хвоста списка, и фрагмент памяти "утечет".

```cpp
void insertAfter(Node* prevNode, int value) {
    if (prevNode == nullptr) {
        // Ошибка или защита от разыменования нулевого указателя
        return;
    }

    Node* newNode = new Node;
    newNode->data = value;

    // БЕЗОПАСНАЯ СШИВКА (Порядок важен!)
    newNode->next = prevNode->next; // 1. Берем старый мост от prevNode и вешаем на newNode
    prevNode->next = newNode;       // 2. prevNode теперь смотрит на newNode
}
```

## 3. Удаление элемента по значению (Delete by value)

Операция удаления требует более аккуратного подхода. Чтобы "отцепить" один узел из цепи, мы обязаны перепрофилировать мост `next` у **предшествующего** узла. То есть мы должны "забегать вперед" зрением, чтобы остановиться за один шаг до жертвы. Мы не можем просто встать на саму жертву и попытаться удалиться — язык C++ не знает, кто привел нас на текущий узел.

```cpp
void deleteValue(Node*& head, int targetValue) {
    if (head == nullptr) return; // Список пуст, удалять нечего

    // ОСОБЫЙ КРАЕВОЙ СЛУЧАЙ: Жертва — это самый первый элемент
    if (head->data == targetValue) {
        Node* nodeToDelete = head;
        head = head->next;    // Корона передается следующему звену
        delete nodeToDelete;  // Освобождаем очищенную память
        return;
    }

    // СТАНДАРТНЫЙ СЛУЧАЙ: Ищем жертву в недрах списка
    Node* current = head;

    // Ищем узел, ПРЕДШЕСТВУЮЩИЙ удаляемому (Забегание вперед)
    while (current->next != nullptr && current->next->data != targetValue) {
        current = current->next;
    }

    // Если цикл закончился по причине тупика (nullptr), цель отсутствует
    if (current->next == nullptr) {
        return;
    }

    // Извлечение и сшивка: current стоит ПРЯМО ПЕРЕД вагоном-жертвой
    Node* nodeToDelete = current->next;
    // Мост строится через крышу жертвы (Сшиваем Концы)
    current->next = nodeToDelete->next;

    // Физическое уничтожение блока
    delete nodeToDelete;
}
```

## Протокол прохождения (Механика удаления из середины)

Проследим алгоритм поиска и удаления узла со значением `15` из списка: `[5] -> [15] -> [99] -> NULL`.

1. Вызов `deleteValue(head, 15)`. `head` указывает на `[5]`. Список не пуст. Равен ли `head->data` (это 5) нашему числу 15? Нет.
2. Бегунок `current` устанавливается на узел `[5]`.
3. Начинается цикл `while`:
   - `current->next` смотрит на узел `[15]` (это не подделка/`nullptr`).
   - Значение `current->next->data` равно `15`. Это совпадает с нашей целью (`targetValue`).
   - Условие цикла `!= targetValue` принимает значение ЛОЖЬ (`false`). Цикл моментально останавливается.
4. Произошла остановка. Указатель `current` сейчас физически указывает на узел `[5]`.
5. Изоляция жертвы: `Node* nodeToDelete = current->next;` — в спасательную переменную сохраняется адрес узла `[15]`.
6. Сшивка: `current->next = nodeToDelete->next;`. Этот шаг берет адрес узла `[99]`, который лежал внутри жертвы `[15]`, и передает его узлу `[5]`. С этого момента узел `[5]` напрямую бьет "струей" указателя в `[99]`. Узел `[15]`, оставшийся без входящих ссылок, превращается в мусор в ОЗУ.
7. Дефрагментация: `delete nodeToDelete;` физически очищает те 16 байт в куче (Heap), которые занимал узел `[15]`.
8. Окончательный вид списка: `[5] -> [99] -> NULL`.

## Резюме характеристик

| Операция                   | Временная сложность         | Логическое описание и особенности                                                                                                                                           |
| :------------------------- | :-------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Поиск (Search)**         | $O(N)$                      | Классический линейный проход. Для нахождения узла в худшем случае требуется проверить каждый элемент (вагон) от начала до конца.                                            |
| **Вставка (Insert After)** | $O(1)$                      | Если локация вставки заранее известна (указатель `prevNode`), то вставка мгновенна. Нет необходимости сдвигать массив данных вправо!                                        |
| **Удаление (Delete)**      | $O(N)$ поиск + $O(1)$ удал. | Процесс состоит из двух фаз: нахождение предшествующего узла и моментальная переброска указателя (сшивка). Крайне важна обработка краевого случая удаления элемента `head`. |

## Контрольные вопросы для самопроверки

- С точки зрения асимптотической сложности (Big-O нотации), в чем заключается принципиальная архитектурная разница между поиском элемента по известному индексу `k` в классическом массиве (`std::vector`) и переходом к `k`-ому элементу в связном списке?
- Почему в функции вставки `insertAfter` операция изменения ссылок строго регламентирована: что произойдет с целостностью структуры данных, если программист ошибочно сначала выполнит `prevNode->next = newNode;`, а только потом попытается зацепить хвост через `newNode->next = prevNode->next;`?
- В функции удаления по значению `deleteValue` мы применяем алгоритм "забегания вперед", проверяя условие `current->next->data != targetValue`. Объясните, почему мы не можем проверять сам `current` (т.е. `current->data != targetValue`) и удалять его, когда найдем? Почему нам критически важно остановиться именно на предшествующем (предыдущем) узле?
- Объясните физический смысл ошибки "Access Violation" (нарушение прав доступа сегментации памяти), которая неизбежно возникнет у пользователя в функции `insertAfter`, если он передаст в аргументы пустой указатель, а программист не напишет защитный блок `if (prevNode == nullptr)`?
- Существует ли сценарий, при котором переменная `head` будет модифицирована в теле функции `deleteValue`? Какое именно сочетание входных параметров должно произойти, чтобы сработал блок `ОСОБЫЙ КРАЕВОЙ СЛУЧАЙ`?
