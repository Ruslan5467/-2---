# Модульное программирование. Структура модульной программы в С++

## Физическая реализация модульности в C++

В языке C++ парадигма модульного программирования реализуется на физическом уровне через разделение исходного кода на различные файлы. В отличие от некоторых современных языков (например, Java или C#, где границами модуля выступают классы и пакеты), классический C++ жестко разделяет **интерфейс** и **реализацию** на уровне файловой системы операционной системы.

Стандартный программный модуль в C++ концептуально состоит из пары файлов:

1.  **Заголовочный файл** (Header file, расширения `.h`, `.hpp` или `.hxx`).
2.  **Файл реализации** (Source file, расширения `.cpp`, `.cc` или `.cxx`).

Кроме того, программа всегда содержит один **Главный файл** (содержащий точку входа — функцию `main()`), который выступает дирижером, объединяющим написанные модули для выполнения конечной задачи.

---

## 1. Заголовочный файл (Интерфейс модуля / Файл `.h`)

Заголовочный файл — это лицо модуля, его **Интерфейс**. Его единственная задача — сообщить компилятору и другим программистам о том, _какие_ инструменты предоставляет данный модуль, не раскрывая того, _как_ они работают.

**Что ДОЛЖНО содержаться в заголовочном файле:**

- Прототипы (объявления) функций.
- Определения пользовательских типов данных (структуры `struct`, перечисления `enum`, классы `class`).
- Внешние объявления глобальных переменных (используя ключевое слово `extern`).
- Подключения других необходимых заголовочных файлов (директивы `#include`).

**Что КАТЕГОРИЧЕСКИ НЕЛЬЗЯ писать в заголовочном файле:**

- Определения (тела) функций (за исключением особых `inline` и шаблонных функций). Если написать реализацию функции в `.h` файле, и подключить этот файл в несколько разных `.cpp` модулей, сборщик программ (Linker) выдаст критическую ошибку "Multiple Definition" (Двойное определение одного и того же символа).

### Защита от двойного включения (Include Guards)

Часто возникает ситуация перекрестных зависимостей: Модуль А подключает Модуль Б. Модуль В подключает Модуль А и Модуль Б. Без защиты компилятор дважды скопирует содержимое `.h` файла Модуля Б, что приведет к ошибке переопределения (рекурсивному зацикливанию).

Для защиты интерфейса используются стражи включения (Include Guards) директивами препроцессора:

```cpp
// Традиционный способ (макросы препроцессора)
#ifndef MATH_UTILS_H    // Если макрос еще не определен...
#define MATH_UTILS_H    // Определить макрос

// ... здесь располагаются прототипы функций ...

#endif // MATH_UTILS_H

// Современный альтернативный способ (поддерживается всеми компиляторами)
#pragma once
```

## 2. Файл реализации (Код модуля / Файл `.cpp`)

Файл реализации — это "черный ящик" модуля. Здесь пишется алгоритмическая логика. Содержимое этого файла скрыто от других частей программы.

**Правила оформления `.cpp` файла:**

1.  **Самоподключение:** Файл реализации ОБЯЗАН первой строчкой подключать свой собственный заголовочный файл (свой интерфейс), чтобы компилятор мог сверить написанные тела функций с заявленными прототипами.
2.  **Реализация:** Здесь располагается технический код (тела подпрограмм).
3.  **Локальность:** Если в `.cpp` файле создать глобальную переменную или вспомогательную функцию и не вынести ее объявление в `.h`, она будет видна _только_ внутри этого файла. Это отличный способ инкапсулировать детали и не загрязнять общее пространство имен.

## Связывание модулей: Директива препроцессора #include

Связь между модулями обеспечивается директивой `#include`.
Важно понимать: `#include` — это не "умная" команда языка программирования. Это команда простому текстовому **препроцессору**. Когда препроцессор видит `#include "math.h"`, он буквально удаляет эту строчку и копипастит (вставляет) вместо нее ВЕСЬ текст из файла `math.h` сплошным полотном прямо перед процессом компиляции.

**Распознавание синтаксиса:**

- `#include <vector>` (в угловых скобках): Команда препроцессору искать файл в стандартных системных директориях компилятора (системные библиотеки).
- `#include "my_math.h"` (в двойных кавычках): Команда искать файл в локальной папке разрабатываемого проекта (собственные модули).

## Пример: Архитектура из трех файлов (Калькулятор)

**1. Модуль Математики: Интерфейс (`math_utils.h`)**

```cpp
#pragma once // Защита от двойного включения

// Интерфейс модуля. Сообщаем, что такие функции существуют.
int add(int a, int b);
int multiply(int a, int b);
```

**2. Модуль Математики: Реализация (`math_utils.cpp`)**

```cpp
#include "math_utils.h" // 1. Обязательно подключаем свой заголовок

// 2. Реализуем "коммерческую тайну" алгоритмов.
int add(int a, int b) {
    return a + b;
}

int multiply(int a, int b) {
    return a * b;
}
```

**3. Главный файл: Точка входа (`main.cpp`)**

```cpp
#include <iostream>      // Системный модуль вывода
#include "math_utils.h"  // НАШ модуль математики (кавычки).
// Обратите внимание: подключается ТОЛЬКО .h файл!
// Файл .cpp подключать (#include "math_utils.cpp") КАТЕГОРИЧЕСКИ запрещено!

int main() {
    std::cout << "Сумма 5 и 3 = " << add(5, 3) << "\n";
    std::cout << "Умножение 5 на 3 = " << multiply(5, 3) << "\n";
    return 0;
}
```

## Резюме характеристик

| Элемент                  | Расширение       | Назначение и содержимое                                                                           |
| :----------------------- | :--------------- | :------------------------------------------------------------------------------------------------ |
| **Заголовочный файл**    | `.h` / `.hpp`    | **Интерфейс**. Прототипы функций, определения структур/классов. Знакомство компилятора с именами. |
| **Файл реализации**      | `.cpp` / `.cc`   | **Реализация**. Математика, циклы, алгоритмы (тела функций). Скрыт от других модулей.             |
| **Защита включения**     | `#pragma once`   | Предотвращает дублирование кода, если `.h` файл случайно подключается несколько раз.              |
| **Связывание (Кавычки)** | `#include "..."` | Поиск файлов интерфейса (собственных модулей) в локальном каталоге разработчика.                  |

## Контрольные вопросы для самопроверки

- Почему в C++ принято разделять модуль на два физических файла: заголовочный `.h` и файл реализации `.cpp`? Какую архитектурную цель преследует это разделение?
- Что такое директива `#include` "под капотом"? Опишите, какие механические действия (с текстом кода) совершает препроцессор C++, когда встречает эту директиву.
- Какую критическую ошибку (на этапе Линкера) совершит начинающий программист, если напишет полное тело (реализацию) функции непосредственно в заголовочном `.h` файле, а затем подключит этот файл к двум разным `.cpp` модулям программы?
- В чем заключается техническое отличие использования угловых скобок (`<iostream>`) от двойных кавычек (`"my_module.h"`) в директиве `#include`?
- Для чего программисты применяют директиву `#pragma once` или традиционную конструкцию `#ifndef ... #define ... #endif` в начале заголовочных файлов? От какой проблемы спасают эти "стражи включения"?
