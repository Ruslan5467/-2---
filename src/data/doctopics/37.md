# Работа с файлами. Файловая система. Общие вопросы, термины и определения

## Теоретическая база: Что такое файл и файловая система?

До этого момента все данные в наших программах (переменные, массивы, деревья) "жили" исключительно в **Оперативной памяти (ОЗУ / RAM)**. Главный недостаток ОЗУ заключается в ее энергозависимости. Как только программа завершается (или отключается электричество), операционная система зачищает процесс, и все вычисленные данные безвозвратно исчезают.

Чтобы сохранить данные между запусками программы или передать их на другой компьютер, информацию необходимо сохранить на **энергонезависимый носитель** (HDD, SSD, Flash-накопитель).

Вся работа с физическими дисками абстрагирована операционной системой. Программисту не нужно знать, в какой физической ячейке твердотельного накопителя находится байт. Вместо этого он оперирует удобными абстракциями: **Файлами** и **Файловой системой**.

- **Файл (File):** Это именованная, логически цельная, непрерывная последовательность байтов на энергонезависимом носителе. По сути, это просто "коробка с битами", у которой есть имя. С точки зрения ОС нет разницы между `.txt` или `.jpg` — это все потоки байтов. Разница заключается лишь в том, _как_ программа будет эти байты интерпретировать.
- **Файловая система (FS):** Это специализированная часть Операционной Системы (такие как NTFS, FAT32, ext4), которая полностью берет на себя тяжелую логистику: она управляет таблицами кластеров, организует файлы в древовидную иерархию (папки/каталоги) и контролирует права доступа к данным.
- **Расширение (Extension):** Часть имени файла, отделенная точкой (например, `report.txt`, `game.exe`). В Windows расширение служит подсказкой для ОС, в какой программе открывать файл. В Linux (Unix-системах) расширения часто носят сугубо декоративный характер — система определяет тип файла по его внутренним заголовкам (Magic numbers).

## Архитектура путей (Path)

Чтобы программа могла получить доступ к файлу на жестком диске, она обязана передать Операционной Системе его точный адрес (координаты). Этот адрес называется **Путь (Path)** к файлу.

В программировании критически важно различать два вида путей:

### 1. Абсолютный (Полный) путь

Это путь, который начинается от самого "корня" файловой системы и прописывает каждый шаг до целевого файла.

- _Особенности:_ Он работает _всегда_, независимо от того, из какой папки была запущена ваша программа.
- _Пример Windows:_ `C:\Users\Admin\Documents\data.txt`
- _Пример Linux:_ `/home/admin/documents/data.txt`

### 2. Относительный путь

Это короткий путь, который вычисляется относительно **Текущей Рабочей Директории (Current Working Directory, CWD)** — то есть папки, откуда был запущен исполняемый `.exe` файл вашей программы (или откуда открыт терминал).

- _Пример:_ `data.txt` (Файл лежит прямо рядом с программой в той же папке).
- _Пример шага глубже:_ `logs\error.txt` (Из папки с программой зайти в папку logs).
- _Пример возврата (прыжок на уровень выше):_ `..\data.txt` (Спецсимвол `..` означает "выйти на одну папку назад (на уровень вверх)").

**Золотое правило:** Практически всегда в коде следует использовать _относительные пути_. Если вы "захардкодите" абсолютный путь `C:\Users\Oleg\...`, ваша программа завершится с ошибкой (Crash) на компьютере любого другого пользователя или на сервере, так как такой папки у них не существует.

## Виды файлов: Текстовые против Бинарных

С точки зрения языков программирования (С/С++), файлы условно делятся на два лагеря по способу чтения и записи:

1. **Текстовые файлы (Text files):**
   - Данные хранятся в виде кодированных символов (ASCII или UTF-8/16).
   - Если открыть такой файл в Блокноте (Notepad), вы увидите читаемый человеком текст (слова, цифры, переносы строк).
   - _Специфика ОС:_ В текстовом режиме C++ автоматически "переводит" спецсимволы. Например, при записи `\n` (перевод строки) на операционной системе Windows он физически запишет два байта: `\r\n` (возврат каретки + сдвиг).

2. **Бинарные / Двоичные файлы (Binary files):**
   - Данные сохраняются ровно в том виде, в каком они лежат в оперативной памяти компьютера (в виде "сырых" байтов: 0 и 1), без каких-либо кодировок в человеческие буквы.
   - Блокнот не сможет их прочитать (он покажет кракозябры или пустые квадраты).
   - _Преимущества:_ Огромная скорость чтения/записи (процессору не нужно переводить числа в символы и обратно). Кроме того, они не подвергаются конвертации переносов строк (ОС ничего не подмешивает). Картинки, музыка `.mp3`, 3D-модели — это всё бинарные файлы.

## Жизненный цикл работы с файлом (The File I/O Lifecycle)

В любом языке программирования (C++, C#, Python, Java) работа с файлами подчиняется строгому 3-шаговому паттерну "Потокового ввода/вывода" (File I/O Stream).

1.  **Открытие файла (Отказ от изоляции ОС):**
    Ваша программа не может просто взять и "прыгнуть" на жесткий диск. Она делает системный вызов к ОС (System call) с просьбой: _"Операционная система, дай мне доступ к файлу `data.txt` в режиме записи"_. ОС проверяет ваши права доступа, находит файл на диске, блокирует его от других программ и выдает вам **Файловый дескриптор (Поток / Stream)** — безопасную "трубу" для передачи байтов.
2.  **Чтение или Запись (Transfer):**
    Программа гоняет байты по выделенной трубе-потоку из ОЗУ на диск или обратно.
3.  **Закрытие файла (Release / Утечка дескрипторов):**
    **Критически важный шаг.** Как только работа завершена, программа ОБЯЗАНА сказать ОС: _"Я закончил, закрой трубу"_. Если этого не сделать (сбой `close()`), возникнут проблема **Resource Leak (Утечка ресурсов)**. Файл останется "заблокированным" для удаления, а так как ОС имеет жесткий лимит на количество одновременно открытых файлов, ваша программа может в итоге обрушиться. Кроме того, команды записи могут остаться в "кэше/буфере" ОЗУ и не дойти до самого диска, если не выполнить закрытие.

## Резюме характеристик

| Понятие                   | Описание в программировании                                                                                                              |
| :------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------- |
| **Файл**                  | Блок памяти на HDD/SSD. Абстрактная коробка с байтами данных.                                                                            |
| **Относительный путь**    | Маршрут доступа, вычисляемый от папки, где физически запущен исполняемый файл приложения (CWD). Идеально для переносимости.              |
| **Бинарный файл**         | "Сырой" дамп памяти из ОЗУ. Нечитаем для глаз в блокноте, но самый быстрый для парсинга центральным процессором ($O(1)$ на конвертацию). |
| **Поток (Stream)**        | Специализированный объект C++, который служит абстракцией (трубой) между программой в ОЗУ и физическим файлом на диске.                  |
| **Обязательное закрытие** | Операция, которая синхронизирует (сбрасывает, `flush`) кэш записи на физический диск и разблокирует файл для ОС.                         |
