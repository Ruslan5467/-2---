# Шаблонные функции

## Теоретическая база: Проблема дублирования кода и Отрицание DRY

Как было рассмотрено ранее, механизм **перегрузки функций** элегантно решает проблему именования: мы можем создать единый интерфейс (одно имя) для работы с разными типами данных.

Однако перегрузка порождает новую, более серьезную архитектурную проблему — **массовое дублирование алгоритмической логики**. Представьте классическую функцию `swap` (обмен значений). Если программе нужно менять местами целые числа, числа с плавающей точкой, символы и строки, программисту придется написать четыре перегруженные функции:
`void swap(int&, int&);`
`void swap(double&, double&);`
`void swap(char&, char&);`
`void swap(std::string&, std::string&);`

Математическая логика и внутренний код этих четырех функций будут абсолютно (символ в символ!) идентичными. Различаются только названия типов данных. Это грубейшее нарушение фундаментального паттерна разработки **DRY (Don't Repeat Yourself — Не повторяйся)**. Если в алгоритме будет найдена ошибка, инженеру придется вручную исправлять ее в десятках перегруженных копий.

Инженерным решением этой проблемы в C++ (и основой всей Стандартной библиотеки шаблонов STL) является парадигма **Обобщенного программирования (Generic Programming)**, реализуемая через **Шаблонные функции (Template Functions)**.

---

## Концепция Шаблона (Чертеж для компилятора)

Шаблонная функция — это **не настоящий исполняемый код**. Это высокоуровневый алгоритмический трафарет (чертеж, лекало).

Программист пишет логику функции только один раз, используя вместо конкретных типов данных (наподобие `int` или `double`) абстрактную переменную-заглушку, которую принято называть `T` (от слова Type). Шаблон указывает компилятору: _"Запомни эту логику. Когда кто-то в коде попытается вызвать эту функцию с конкретными данными, пожалуйста, сам подставь их тип вместо буквы `T` и сгенерируй для меня готовую функцию"_.

---

## Архитектурные особенности и Синтаксис

Для создания шаблона используется ключевое слово `template`, за которым в угловых скобках `< >` перечисляются абстрактные параметры (типы-заглушки) с префиксом `typename` или `class` (в данном контексте эти слова абсолютно эквивалентны).

**Синтаксис объявления:**

```cpp
// "Мы объявляем шаблон, в котором T будет выступать в роли имени типа данных"
template <typename T>
T findMax(const T& a, const T& b) {
    if (a > b) return a;
    return b;
}
```

В этом примере `T` универсален. Он выступает и как тип возвращаемого значения, и как тип константных ссылок на передаваемые аргументы.

## Пример использования (Универсальность в действии)

Реализуем обобщенную функцию поиска максимума и вызовем ее для совершенно разных типов данных.

```cpp
#include <iostream>
#include <string>

// ОПРЕДЕЛЕНИЕ ШАБЛОНА
template <typename T>
T getMaximum(const T& x, const T& y) {
    std::cout << "[Template] Сравнение объектов...\n";
    return (x > y) ? x : y;
}

int main() {
    // 1. Вызов для типа int
    // Компилятор видит сигнатуру (int, int). Он сам догадывается, что T = int.
    int maxInt = getMaximum(42, 10);
    std::cout << "Максимум из целых: " << maxInt << "\n\n";

    // 2. Вызов для типа double
    // Компилятор видит (double, double) и подставляет T = double.
    double maxDouble = getMaximum(3.14, 9.81);
    std::cout << "Максимум из дробных: " << maxDouble << "\n\n";

    // 3. Вызов для типа std::string
    // Шаблоны работают с классами ООП! (если класс поддерживает ператор '>')
    std::string s1 = "Apple";
    std::string s2 = "Zebra";
    std::string maxStr = getMaximum(s1, s2);
    std::cout << "Максимум из строк: " << maxStr << "\n\n";

    return 0;
}
```

## Протокол прохождения (Метапрограммирование и Инстанцирование)

Самая важная архитектурная тайна шаблонов заключается в механике работы компилятора под капотом (во время трансляции кода). Шаблон — это инструмент метапрограммирования ("программы, которая пишет программу").

Что происходит, когда компилятор C++ встречает вызов `getMaximum(42, 10)`?

1.  **Вывод типов (Type Deduction):** Компилятор заглядывает в переданные аргументы. `42` — это целое число. `10` — это целое число. Следовательно, абстрактный типпараметр `T` должен быть жестко привязан к типу `int`.
2.  **Инстанцирование (Instantiation):** Компилятор ставит сборку на паузу. Он берет ваш шаблонный чертеж, удаляет из него все буквы `T`, вписывает вместо них слово `int` и **скрытно (в памяти) генерирует абсолютно новую, обычную функцию** на классическом C++:
    ```cpp
    // Сгенерированный компилятором невидимый код (Инстанс):
    int getMaximum(const int& x, const int& y) { ... }
    ```
3.  **Компиляция в машинный код:** Только теперь эта сгенерированная функция транслируется в физический машинный код (`.exe`) и вызывается.
4.  Встречая дальше вызов `getMaximum(3.14, 9.81)`, компилятор повторяет процесс и генерирует **ВТОРУЮ** физическую функцию `double getMaximum(const double&, const double&)` и вшивает ее в бинарный файл.

### Следствия механизма Инстанцирования:

- **Иллюзия размера:** Если вы написали шаблон, но **ни разу его не вызвали** в функции `main`, компилятор даже не попытается перевести его в машинный код. В итоговом приложении эта функция просто не будет существовать. Шаблон без применения ничего не весит в ОЗУ.
- **Раздувание кода (Code Bloat):** Обратная сторона медали. Если вы вызовете свой шаблон с 20-ю разными типами данных (от `int` до пользовательских классов), компилятор сгенерирует 20 абсолютно уникальных перегруженных функций. Размер исполняемого файла (`.exe`) раздуется.

## Проблема конфликта вывода типов (Deduction Failure)

Так как компилятор C++ действует по принципу жесткой статической типизации, он не умеет "додумывать" типы, если переданные аргументы конфликтуют с шаблоном.

Представим вызов для нашего написанного шаблона `T getMaximum(T x, T y)`:

```cpp
auto result = getMaximum(5, 3.14); // ОШИБКА КОМПИЛЯЦИИ!
```

**Анализ компилятора:**

1. Первый аргумент `5` - это `int`. Значит `T` должно быть `int`.
2. Второй аргумент `3.14` - это `double`. Значит `T` должно быть `double`.
3. Типы-кандидаты противоречат друг другу (`T` не может быть одновременно двумя разными вещами). Происходит _Deduction Failure_.

**Решение проблемы:** Дать компилятору явную команду (Explicit Specification), кто здесь главный, передав требуемый тип в угловых скобках при вызове:

```cpp
// Мы жестко приказываем: T - это double.
// Число 5 будет неявно приведено компилятором к 5.0 (double). Успех!
auto result = getMaximum<double>(5, 3.14);
```

## Резюме характеристик

| Характеристика                 | Описание                                                                                                         |
| :----------------------------- | :--------------------------------------------------------------------------------------------------------------- |
| **Суть механизма**             | "Трафарет" для автоматической генерации функций компилятором                                                     |
| **Решаемая проблема**          | Ликвидация дублирования кода (поддержка принципа DRY)                                                            |
| **Синтаксис параметра типа**   | `template <typename T>` или `template <class T>`                                                                 |
| **Инстанцирование**            | Процесс физического создания компилятором функции из шаблона под конкретный тип                                  |
| **Наличие в памяти (Runtime)** | Сам "шаблон" в ОЗУ не существует. В процессоре выполняются только конкретные сгенерированные インстансы (копии). |
| **Потенциальный риск**         | _Code Bloat_ (раздувание объема бинарного файла при множественных вызовах разных типов)                          |
