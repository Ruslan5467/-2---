# Функции, определяемые пользователем. Структура, пример, протокол прохождения

## Теоретическая база и основные понятия

Современные программные комплексы могут насчитывать миллионы строк кода. Писать такой объем инструкций сплошным неразрывным полотном (внутри одной функции `main`) физически невозможно и архитектурно бессмысленно. Для решения этой проблемы применяется парадигма **процедурной абстракции**, главным инструментом которой выступают **функции, определяемые пользователем (User-Defined Functions)**.

**Функция** — это именованный, логически целостный и обособленный фрагмент программного кода, который принимает входные данные (параметры), выполняет строго определенную последовательность действий над ними и, как правило, возвращает результат вычислений обратно в вызывающий код.

### Инженерные преимущества использования функций:

1.  **Повторное использование кода (Code Reusability):** Принцип DRY (Don't Repeat Yourself). Если алгоритм поиска корней уравнения используется в десяти разных местах программы, он пишется один раз в виде функции, а затем лишь вызывается по имени.
2.  **Модульность (Пошаговая детализация):** Как предписывает метод нисходящего проектирования, сложная задача разбивается на иерархию простых подзадач-функций. Каждую функцию можно разрабатывать, тестировать и отлаживать изолированно от остальной системы.
3.  **Инкапсуляция и сокрытие сложности:** Программисту, использующему функцию `calculatePhysics()`, не обязательно знать ее внутренние математические формулы. Ему достаточно знать ее **интерфейс** — какие данные подать на вход и что ожидается на выходе.
4.  **Уменьшение объема памяти:** Компилятор генерирует машинный код функции только один раз. Множественные вызовы лишь переводят счетчик команд процессора на стартовый адрес этой функции.

## Анатомия и структура функции в C++

С точки зрения компилятора C++, архитектура любой определяемой пользователем функции жестко разделяется на две составляющие: **Прототип (Объявление)** и **Определение (Тело реализации)**.

### 1. Прототип (Сигнатура / Интерфейс)

Прототип информирует компилятор о существовании функции еще до того, как компилятор "увидит" ее внутренности. Это необходимо для строгой статической типизации: если программист попытается передать в функцию текст вместо числа, компилятор на этапе трансляции выдаст ошибку, сверившись с прототипом.

**Синтаксис прототипа:**
`[тип_возвращаемого_значения] [имя_функции]([список_формальных_параметров_с_типами]);`

- **Тип возвращаемого значения:** Любой встроенный тип (`int`, `double`, `bool`), структура или указатель. Если функция ничего не возвращает (выполняет лишь действия), используется ключевое слово `void` (пустота).
- **Имя функции:** Идентификатор (например, `findMax`). Должен отражать суть действия (использовать глаголы — хорошая практика).
- **Список параметров:** Перечень "входных шлюзов", разделенных запятыми, с указанием их типов.

### 2. Определение (Тело функции)

Содержит саму логику (алгоритмические структуры). Заключается в фигурные скобки `{ }`. Обязано заканчиваться оператором `return` (кроме функций типа `void`), который передает вычисленный результат обратно в точку вызова.

## Пример реализации

Рассмотрим классическую задачу: вычислить гипотенузу прямоугольного треугольника по двум катетам, используя теорему Пифагора.

```cpp
#include <iostream>
#include <cmath> // Для функции sqrt()

// --- 1. ПРОТОТИП (Объявление) ---
// Говорит компилятору: "В программе есть подпрограмма calcHypotenuse,
// она ждет два числа с плавающей точкой и вернет одно такое же число".
double calcHypotenuse(double a, double b);

int main() {
    double leg1 = 3.0;
    double leg2 = 4.0;

    std::cout << "Вычисление гипотенузы для катетов " << leg1 << " и " << leg2 << "\n";

    // --- 2. ВЫЗОВ ФУНКЦИИ ---
    // leg1 и leg2 - это ФАКТИЧЕСКИЕ параметры (аргументы) вызова.
    // Выполнение main() приостанавливается, управление передается в функцию.
    double result = calcHypotenuse(leg1, leg2);

    std::cout << "Результат: " << result << std::endl;
    return 0;
}

// --- 3. ОПРЕДЕЛЕНИЕ (Реализация) ---
// a и b - это ФОРМАЛЬНЫЕ параметры (локальные переменные функции).
double calcHypotenuse(double a, double b) {
    // Вычисление квадратов
    double squareA = a * a;
    double squareB = b * b;

    // Внутренняя логика обработки...
    double sumOfSquares = squareA + squareB;

    // Возврат результата и уничтожение локального стека функции
    return std::sqrt(sumOfSquares);
}
```

## Протокол прохождения контрольного примера (Аппаратный стек)

Чтобы понять, как функции работают "под капотом", необходимо рассмотреть механизм аппаратного стека вызовов (Call Stack).

**Контрольный пример:** Программа запускается. `main()` определяет локальные переменные `leg1 = 3.0` и `leg2 = 4.0`. Доходим до строки вызова: `double result = calcHypotenuse(leg1, leg2);`.

Проследим аппаратный протокол выполнения процессором (архитектура x86/x64):

### Шаг 1. Формирование ફрейма в точке вызова (Caller)

- **Сохранение контекста:** Процессор запоминает "адрес возврата" (адрес следующей машинной инструкции в функции `main`, куда нужно будет вернуться после вычисления). Этот адрес проталкивается (push) в системный стек.
- **Передача аргументов:** Копии значений фактических параметров (`3.0` и `4.0`) помещаются в регистры процессора или так же проталкиваются в стек, чтобы функция могла до них добраться.

### Шаг 2. Передача управления (Jump)

- Процессор выполняет инструкцию безусловного перехода `CALL` на адрес начала функции `calcHypotenuse`.

### Шаг 3. Инициализация фрейма функции (Callee)

- Процессор сдвигает регистр указателя стека (Stack Pointer), выделяя новый изолированный блок памяти — **фрейм вызова (Stack Frame)** специально для `calcHypotenuse`.
- Формальные параметры `a` и `b` инициализируются переданными копиями (`a = 3.0`, `b = 4.0`).
- Внутри фрейма выделяется память под локальные переменные `squareA`, `squareB`, `sumOfSquares`.

### Шаг 4. Выполнение тела

- Производятся математические расчеты.
  - `squareA = 9.0`
  - `squareB = 16.0`
  - `sumOfSquares = 25.0`
- Вызывается библиотечная функция `std::sqrt(25.0)`, которая (пройдя аналогичный протокол) возвращает `5.0`.

### Шаг 5. Уничтожение фрейма и Возврат (Return)

- Инструкция `return 5.0;` кладет итоговое значение в специальный регистр процессора (например, `RAX` в x64 архитектуре), через который `main()` сможет его забрать.
- **Очистка Кулисы:** Указатель стека сдвигается обратно. Весь фрейм функции `calcHypotenuse` вместе с переменными `a`, `b`, `squareA` и т.д. объявляется недействительным (память очищается мгновенно `O(1)`).
- Процессор извлекает из стека "адрес возврата" (сохраненный на Шаге 1) и выполняет инструкцию `RET` (Return).

### Шаг 6. Возобновление работы вызывающего кода

- Функция `main()` "просыпается". Она забирает число `5.0` из регистра процессора и присваивает его своей локальной переменной `result`. Выполнение программы продолжается линейно.

## Резюме характеристик

| Характеристика                | Значение / Специфика                                                                        |
| :---------------------------- | :------------------------------------------------------------------------------------------ |
| **Синонимы**                  | Подпрограмма, Процедура, Метод (в ООП)                                                      |
| **Обязательные части**        | Тип возврата, имя, параметры, тело                                                          |
| **Локальность данных**        | Переменные внутри `{}` не видны снаружи (сокрытие контекста)                                |
| **Механизм памяти**           | Для каждого вызова создается новый фрейм на Stack                                           |
| **Стоимость (Overhead)**      | Вызов функции несет мизерные накладные расходы на работу со стеком и копирование аргументов |
| **Жизненный цикл переменных** | Рождаются при входе, умирают при `return`                                                   |
