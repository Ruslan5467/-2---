# Способы обхода бинарного дерева. Прямой обход (preOrder)

## Теоретическая база: Задачи обхода дерева

Обход (Traversal) структуры данных — это систематический процесс посещения каждого узла в этой структуре ровно один раз.

Если в линейных структурах (массивы, списки) обход тривиален (слева направо от начала до конца), то в нелинейных структурах — деревьях — у нас на каждом узле есть выбор: пойти налево или пойти направо. Из-за этого существует несколько устоявшихся стратегий (алгоритмов) обхода древовидных структур.

Основные алгоритмы обхода делятся на две категории:

1. **Поиск в глубину (Depth-First Search, DFS):** Мы идем вниз по ветке до самого упора (до листьев), прежде чем вернуться и исследовать соседние ветки. Сюда относятся три классических рекурсивных обхода: Прямой (Pre-order), Симметричный (In-order) и Обратный (Post-order).
2. **Поиск в ширину (Breadth-First Search, BFS):** Мы исследуем узлы горизонтально, слой за слоем.

### Прямой обход (Pre-order Traversal)

**Прямой обход** относится к категории "Поиск в глубину". Его алгоритмическая формула звучит как **N-L-R** (Node-Left-Right, Узел-Лево-Право).

Правила обхода (строгий порядок):

1. **N (Node):** Сначала "посещаем" (обрабатываем/выводим на экран) сам текущий **Корень/Узел**.
2. **L (Left):** Затем рекурсивно обходим всё **Левое поддерево**.
3. **R (Right):** Затем рекурсивно обходим всё **Правое поддерево**.

Почему он называется прямым? Потому что родительский узел (Корень) обрабатывается (посещается) _прямо сразу_, ДО того, как алгоритм спустится к его детям.

## Пример использования (Сборка дерева и прямой обход)

Напишем функцию `preOrder`, которая применяет данный алгоритм для вывода в консоль значений узлов.

```cpp
#include <iostream>

struct TreeNode {
    int data;
    TreeNode* left;
    TreeNode* right;

    TreeNode(int val) : data(val), left(nullptr), right(nullptr) {}
};

// Функция ПРЯМОГО ОБХОДА (Pre-order: Узел -> Лево -> Право)
void preOrder(TreeNode* node) {
    // Базовый случай (дно рекурсии)
    if (node == nullptr) {
        return;
    }

    // N: Обработка текущего узла (сначала печатаем родителя)
    std::cout << node->data << " ";

    // L: Спускаемся в левое поддерево
    preOrder(node->left);

    // R: Спускаемся в правое поддерево
    preOrder(node->right);
}

// Вспомогательная функция очистки памяти (чтобы не было утечек)
// Спойлер: удаление всегда происходит "Обратным обходом" (Post-order)
void deleteTree(TreeNode* node) {
    if (node == nullptr) return;
    deleteTree(node->left);
    deleteTree(node->right);
    delete node;
}

int main() {
    // Собираем следующее бинарное дерево:
    //          ( 1 )
    //         /     \
    //      ( 2 )   ( 3 )
    //      /   \       \
    //   ( 4 ) ( 5 )   ( 6 )

    TreeNode* root = new TreeNode(1);
    root->left = new TreeNode(2);
    root->right = new TreeNode(3);

    root->left->left = new TreeNode(4);
    root->left->right = new TreeNode(5);

    root->right->right = new TreeNode(6);

    std::cout << "Результат прямого обхода (Pre-order):\n";
    preOrder(root);
    // Ожидаемый вывод: 1 2 4 5 3 6
    std::cout << "\n";

    deleteTree(root);

    return 0;
}
```

## Протокол прохождения (Механика рекурсии `N-L-R`)

Рассмотрим, как физически выстраивается путь обхода нашего дерева в консоли, шаг за шагом:

1. **Корень 1:** Вызываем `preOrder(root)`. Алгоритм видит корень (1). Выполняется шаг **N**: печатаем `1`. Дальше шаг **L**: уходим глубоко влево от корня.
2. **Узел 2:** Становится новым "локальным корнем". Выполняется шаг **N**: печатаем `2`. Шаг **L**: уходим дальше влево от двойки.
3. **Узел 4:** Локальный корень. Шаг **N**: печатаем `4`. Шаг **L**: левого ребенка нет (`nullptr`), возврат. Шаг **R**: правого ребенка нет (`nullptr`), возврат. Узел 4 полностью завершил свою работу! Мост сворачивается назад к узлу 2.
4. **Возврат к Узлу 2:** Узел 2 уже был напечатан и левая ветвь пройдена (узел 4 готов). У него остался шаг **R**. Переходим на правого ребенка — узел 5.
5. **Узел 5:** Локальный корень. Шаг **N**: печатаем `5`. Шаг **L** и **R** — пусто (возврат). Возвращаемся к узлу 2.
6. **Полный возврат узла 2 к Главном Корню:** Узел 2 полностью отработал (сам напечатан, лево напечатано, право напечатано). Он рапортует об успехе Главному Корню (1).
7. Главный Корень (1) завершил шаг **L** (пройдена вся левая половина дерева). Теперь он выполняет свой мощный шаг **R**: переходит в правое поддерево к узлу 3.
8. **Узел 3:** Шаг **N**: печатаем `3`. Шаг **L**: пусто. Шаг **R**: переходим к 6.
9. **Узел 6:** Печатается `6`. Лево и право пусто. Функция завершается.

**Итоговый трек вывода:** `1 -> 2 -> 4 -> 5 -> 3 -> 6`.
Главная особенность, которую можно заметить невооруженным глазом — самый первый логический узел дерева (Главный Корень) обрабатывается и выводится самым ПЕРВЫМ, а далее вывод смещается в самую глубокую левую часть.

## Практическое применение прямого обхода

В Computer Science прямой обход — это не просто абстрактная математика. Он имеет жестко заданные производственные сценарии использования:

1. **Клонирование (Копирование) дерева:** При создании идентичной копии дерева в оперативной памяти вам необходимо сначала скопировать родительский узел (Корень), а затем прикрепить к нему скопированные ветки-потомки. Это можно сделать исключительно прямым обходом (N-L-R).
2. **Сериализация дерева:** Чтобы сохранить (записать) древовидную структуру в плоский бинарный файл или JSON на жестком диске для последующего восстановления, данные первыми сохраняются в виде "Корень -> Дети".
3. **Префиксная нотация в парсерах выражений:** В компиляторах, если синтаксическое дерево (AST) представляет собой математическое выражение, прямой обход выведет его в префиксной форме записи (Польская нотация), например: `+ * A B C`.

## Резюме характеристик

| Характеристика                        | Описание                                                                                                                                           |
| :------------------------------------ | :------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Категория алгоритма**               | DFS (Depth-First Search - Поиск в глубину).                                                                                                        |
| **Поведение (Формула)**               | **N-L-R** (Node, Left, Right). Сначала обрабатывается сам узел, затем рекурсивно заливается левая ветвь, затем правая.                             |
| **Асимптотическая сложность (Время)** | $O(N)$, где $N$ — количество узлов в дереве. Мы посещаем каждый узел ровно 1 раз.                                                                  |
| **Сложность по памяти**               | $O(H)$, где $H$ — высота дерева. На максимальной глубине в стеке вызовов (Call Stack) лежит количество ссылок, равное глубине самой длинной ветки. |
| **Применение**                        | Клонирование дерева (Deep Copy), сериализация для хранения, генерация префиксной нотации (AST парсеры).                                            |

## Контрольные вопросы для самопроверки

- Как расшифровывается аббревиатура N-L-R в контексте прямого обхода (pre-order), и какая логическая последовательность действий скрывается за каждой из этих букв?
- Обоснуйте алгоритмическую причину, по которой прямой обход относится к категории "Поиск в глубину" (DFS), а не к поиску "в ширину" (BFS). Если использовать дерево из примера, какие элементы были бы выведены первыми при поиске в ширину (по слоям)?
- Представьте себе бинарное дерево, которое полностью вырождено (деградировало) в правую сторону (каждый узел имеет только правого потомка). Совпадет ли порядок вывода элементов при прямом обходе с порядком вывода обычного линейного связного списка? Почему?
- Объясните (с точки зрения рекурсии и родительских зависимостей), почему алгоритм _клонирования_ (копирования объекта) дерева невозможно реализовать обходом типа "сначала левое поддерево, затем текущий узел" (in-order)? Почему нам критически важно сначала "обработать" (создать новый инстанс в ОЗУ) сам корень?
- В функции `preOrder` есть базовый случай: `if (node == nullptr) return;`. Что именно (на уровне аппаратного стека или ошибок ОЗУ) произойдет с программой, если мы забудем скомпилировать эту защитную строку, и попытаемся выполнить алгоритм `node->left` на листе (т.е. на узле, у которого дети — это нуль-поинтеры)?
