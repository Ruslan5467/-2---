# Динамические массивы. Одномерные динамические массивы

## Теоретическая база: Стек против Кучи (Stack vs Heap)

Для понимания необходимости динамических массивов необходимо разобраться, как программа на языке C++ взаимодействует с оперативной памятью компьютера. Адресное пространство выполняемого процесса логически разделено на несколько сегментов, два из которых критически важны для хранения данных:

1.  **Стек (Stack):** Область памяти, используемая для выделения локальных переменных внутри блоков кода (функций). Архитектура стека работает по принципу LIFO (Last-In-First-Out). Стек невероятно быстр, так как выделение памяти сводится к простому сдвигу регистра процессора (Stack Pointer). Однако стек имеет два критических ограничения:
    - Жестко лимитированный объем (обычно несколько мегабайт, переполнение вызывает фатальный сбой _Stack Overflow_).
    - **Статичность размеров:** Размер любых массивов, выделяемых на стеке (например, `int arr[100];`), должен быть точно известен компилятору еще **до** запуска программы (на этапе трансляции).

2.  **Динамическая память / Куча (Heap или Free Store):** Глобальный пул оперативной памяти, управляемый операционной системой. Объем кучи ограничен лишь доступной физической RAM и файлом подкачки. Главная особенность: память здесь выделяется _адаптивно_, непосредственно во время выполнения программы (runtime).

**Главная архитектурная проблема:** В реальных инженерных задачах размер обрабатываемых данных практически никогда не известен заранее. Программа может считывать записи из базы данных, загружать изображение скачанного файла или ожидать пользовательского ввода. Использование статических массивов в таких случаях приводит либо к неоправданному перерасходу памяти при выделении "с запасом", либо к падению программы при нехватке места. Разрешением этого фундаментального конфликта и выступают **динамические массивы**.

## Одномерные динамические массивы в C++

Одномерный динамический массив — это структура данных, представляющая собой непрерывный блок оперативной памяти, размер которого запрашивается у операционной системы в процессе работы программы.

### Жизненный цикл динамического массива:

1.  **Запрос (Аллокация):** Осуществляется с помощью оператора `new[]`. Программа просит операционную систему найти цельный блок памяти требуемого размера.
2.  **Инициализация указателя:** Оператор `new[]` возвращает адрес первого байта выделенного блока. Этот адрес присваивается указателю (pointer). Без указателя программа мгновенно «потеряет» выделенную память.
3.  **Эксплуатация:** Работа с динамическим массивом абсолютно идентична работе со статическим — через индексацию `[ ]` или арифметику указателей `*(ptr + i)`.
4.  **Уничтожение (Освобождение):** Самый критичный этап. В отличие от стека, куча не очищается автоматически при выходе из функции. Разработчик обязан самостоятельно вернуть запрошенную память операционной системе с помощью оператора `delete[]`.

### Фатальные ошибки архитектуры ручного управления памятью:

- **Утечка памяти (Memory Leak):** Возникает, если программист забыл вызвать `delete[]` или указатель был перезаписан/утерян. Зарезервированный блок остается "занятым" с точки зрения ОС (пока программа не завершится), что в серверных приложениях (работающих 24/7) неуклонно ведет к истощению RAM и аварийному падению (OOM, Out Of Memory).
- **Висячий указатель (Dangling Pointer):** Ситуация, когда память уже очищена оператором `delete[]`, но указатель все еще хранит старый адрес. Попытка обратиться по этому адресу приведет к неопределенному поведению (Undefined Behavior) или мгновенной аварии (Segmentation Fault).

## Пример реализации и синтаксис

Ниже представлен классический сценарий создания одномерного динамического массива целочисленных значений (int), его заполнения и последующего безопасного уничтожения.

```cpp
#include <iostream>

void dynamicArrayExample() {
    int arrSize;

    // 1. Определение размера массива во время выполнения (Runtime)
    std::cout << "Введите размер массива (например, 5): ";
    std::cin >> arrSize;

    if (arrSize <= 0) {
        std::cerr << "Ошибка: Размер массива должен быть положительным!" << std::endl;
        return;
    }

    // 2. ВЫДЕЛЕНИЕ ПАМЯТИ (Аллокация)
    // ОС находит блок размером arrSize * sizeof(int) (например, 5 * 4 = 20 байт)
    // и возвращает адрес его начала в указатель pArray.
    int* pArray = new int[arrSize];

    // 3. ЭКСПЛУАТАЦИЯ: Инициализация и доступ по индексу
    for (int i = 0; i < arrSize; ++i) {
        pArray[i] = (i + 1) * 10; // Заполняем: 10, 20, 30, 40, 50...
    }

    // 4. ЭКСПЛУАТАЦИЯ: Арифметика указателей (альтернативный синтаксис)
    std::cout << "Содержимое динамического массива: ";
    for (int i = 0; i < arrSize; ++i) {
        // *(pArray + i) логически эквивалентно pArray[i]
        std::cout << *(pArray + i) << " ";
    }
    std::cout << std::endl;

    // 5. ОСВОБОЖДЕНИЕ ПАМЯТИ (Критически важный шаг!)
    // Квадратные скобки [] обязательны, они сигнализируют ОС,
    // что нужно удалить весь массив, а не только один элемент.
    delete[] pArray;

    // 6. ПРАВИЛО ХОРОШЕГО ТОНА: Обнуление указателя (защита от dangling pointer)
    pArray = nullptr;
}

int main() {
    dynamicArrayExample();
    return 0;
}
```

## Протокол прохождения контрольного примера

Контрольный пример: Пользователь вводит `arrSize = 3`. Проследим за состоянием памяти шаг за шагом.

- **Шаг 1: `cin >> arrSize;`**
  - _Стек:_ Создается переменная `arrSize` (4 байта), ей присваивается значение `3`.
- **Шаг 2: `int* pArray = new int[arrSize];`**
  - _Куча (Heap):_ Диспетчер памяти выделяет непрерывный блок в `3 * sizeof(int) = 12 байт`. Предположим, начальный адрес блока `0x1A40`.
  - _Стек:_ Создается переменная `pArray` (обычно 8 байт на 64-битн. системах), в которую записывается значение адреса `0x1A40`.
- **Шаг 3: Цикл инициализации (Итерации 0, 1, 2)**
  - _i=0:_ Запись значения `10` по адресу `0x1A40`.
  - _i=1:_ Запись значения `20` по адресу `0x1A44` (адрес + 4 байта, так как тип int).
  - _i=2:_ Запись значения `30` по адресу `0x1A48`.
- **Шаг 4: `delete[] pArray;`**
  - _Куча (Heap):_ Диспетчер памяти помечает 12 байт, начиная с адреса `0x1A40`, как "свободные" (возвращает их в пул доступной памяти системы). Данные (10, 20, 30) внутри могут остаться как "мусор", но они больше нам не принадлежат.
  - _Стек:_ Указатель `pArray` **все еще** хранит значение `0x1A40` (висячий указатель!).
- **Шаг 5: `pArray = nullptr;`**
  - _Стек:_ Значение `pArray` затирается нулем (`0x0000`). Теперь обращение к нему безопасно прервет программу с четкой ошибкой (Null Pointer Exception), а не приведет к непредсказуемому разрушению других данных в куче. Защита сработала.

## Резюме характеристик (Одномерный динамический массив)

| Характеристика                           | Значение / Оценка                                                        |
| :--------------------------------------- | :----------------------------------------------------------------------- |
| **Расположение в памяти**                | Динамическая память (Heap/Куча)                                          |
| **Непрерывность памяти**                 | Строго непрерывный блок байтов (High Cache Locality)                     |
| **Сложность: Доступ по индексу**         | `O(1)` (Арифметика сдвига адреса выполняется мгновенно)                  |
| **Сложность: Вставка/удаление в конец**  | Невозможно без создания нового массива. Для реаллокации требуется `O(N)` |
| **Сложность: Вставка в начало/середину** | Невозможно без реаллокации `O(N)` и сдвига элементов `O(N)`              |
| **Оператор выделения**                   | `new[]`                                                                  |
| **Оператор уничтожения**                 | `delete[]`                                                               |

## Контрольные вопросы для самопроверки

- Чем архитектурно и функционально сегмент памяти "Стек" отличается от сегмента "Куча"?
- Почему при удалении динамического массива обязательно использовать оператор `delete[]` с квадратными скобками, а не просто `delete`? Что произойдет в случае ошибки?
- Что такое утечка памяти (memory leak) и к каким последствиям она приводит в долго работающих программах (например, веб-серверах)?
- Дайте определение "висячему указателю" (dangling pointer). Почему присваивание указателю значения `nullptr` после очистки памяти считается лучшей инженерной практикой?
- В чем заключается проблема производительности при изменении (расширении) размера уже существующего динамического массива?
