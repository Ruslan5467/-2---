# Основные операции с односвязными линейными списками со статическим головным элементом

## Теоретическая база: Стек против Кучи (Stack vs Heap)

Рассматривая паттерн "списка с фиктивной (dummy) головой" в предыдущей лекции, мы выделяли память под фиктивно-пустой блок динамически в Куче (`Node* dummyHead = new Node;`). Но так как этот служебный узел-заглушка создается один раз при старте, его данные не меняются, и он живет ровно до разрушения самого списка, существует альтернативный, более быстрый системный подход: **Использование статического головного элемента**.

**Суть статической фиктивной головы:**
Вместо того чтобы просить операционную систему искать память в Куче и выдавать нам указатель (`Node*`), мы создаем узел напрямую как обычную локальную или глобальную переменную в Стеке вызовов (Stack).

Пример в функции `main`:

```cpp
Node staticDummyHead;          // Объект создается прямо в стеке (работает мгновенно)
staticDummyHead.data = 0;      // Фиктивные данные
staticDummyHead.next = nullptr;// Список пока пуст
```

При такой архитектуре нам не нужно беспокоиться о случайных утечках памяти при удалении самой фиктивной головы — C++ уничтожит переменную `staticDummyHead` автоматически при выходе из области видимости (например, при завершении `main`). Главной особенностью становится то, что во все функции мы теперь передаем не указатель `Node* head`, а сам объект фиктивной головы **по ссылке** `Node& head`.

---

## 1. Вставка в начало (Push Front)

Так как сама фиктивная голова лежит на стеке (это объект, а не указатель), доступ к ее полям осуществляется через оператор `.` (точку), а не через стрелку `->`. Но все создаваемые реальные узлы с данными (для гибкости) по-прежнему выделяются динамически через `new`.

```cpp
#include <iostream>

struct Node {
    int data;
    Node* next;
};

// Функция принимает фиктивную голову ПО ССЫЛКЕ (Node&),
// чтобы изменять оригинальный мост next, принадлежащий этой голове на стеке.
void insertFront(Node& dummyHead, int value) {
    // 1. Выделяем память под реальные данные в Куче!
    Node* newNode = new Node;
    newNode->data = value;

    // 2. Сшиваем вагоны (Новый узел цепляется за старый первый элемент ОЗУ)
    // Обратите внимание на ТОЧКУ (dummyHead.next)
    newNode->next = dummyHead.next;

    // 3. Статическая голова направляет свой вектор на свежесозданный узел
    dummyHead.next = newNode;
}
```

## 2. Удаление первого вхождения по значению (Delete by value)

Алгоритм поиска предшественника (забегания вперед) сохраняется в первозданном виде. Главное отличие — правильная инициализация стартовой позиции бегунка. Поскольку наш "бегунок" — это указатель (`Node*`), нам нужно взять адрес переменной `dummyHead`, находящейся на стеке, с помощью оператора взятия адреса `&`.

```cpp
void deleteValue(Node& dummyHead, int targetValue) {
    // Запускаем бегунка. Оператор & обязателен, чтобы получить из объекта УКАЗАТЕЛЬ на него
    Node* current = &dummyHead;

    // Забегающий цикл while
    while (current->next != nullptr && current->next->data != targetValue) {
        current = current->next; // Прыжки по динамической памяти (->)
    }

    // Если уперлись в тупик — не нашли
    if (current->next == nullptr) {
        return;
    }

    // Изоляция жертвы и сшивка разорванной цепи
    Node* nodeToDelete = current->next;
    current->next = nodeToDelete->next;

    // Фрагментация (Очистка узла в Куче)
    delete nodeToDelete;
}
```

## 3. Печать и Очистка (Специфика деструктуризации)

При очистке списка со статическим фиктивным узлом мы **ни в коем случае нельзя удалять (delete) саму фиктивную голову**. Оператор `delete` предназначен ТОЛЬКО для памяти, выделенной через `new` (Кучу). Попытка вызвать `delete` для переменной на стеке приведет к обрушению программы (Crash / Segfault). Поэтому метод очистки очищает только цепочку `next`.

```cpp
// Печать элементов
void printList(const Node& dummyHead) {
    std::cout << "Список: ";
    // Начинаем парсинг с первого ДИНАМИЧЕСКОГО элемента
    Node* current = dummyHead.next;

    while (current != nullptr) {
        std::cout << current->data << " -> ";
        current = current->next;
    }
    std::cout << "NULL\n";
}

// Очистка памяти (Уничтожаем только полезную нагрузку!)
void clearList(Node& dummyHead) {
    Node* current = dummyHead.next; // Встаем на первый динамический элемент

    while (current != nullptr) {
        Node* temp = current;
        current = current->next;
        // Уничтожаем динамический узел! (Сама dummyHead остается невредимой на стеке)
        delete temp;
    }

    // Сбрасываем указатель пустого списка на 0
    dummyHead.next = nullptr;
}
```

## Архитектурное сравнение и применение (Сборка в main)

Как все эти функции сливаются воедино при разработке реального приложения?

```cpp
int main() {
    // 1. Инициализация (МГНОВЕННО, на системном стеке приложения)
    Node myDummyHead;
    myDummyHead.next = nullptr;

    // 2. Эксплуатация
    insertFront(myDummyHead, 100);
    insertFront(myDummyHead, 200);
    insertFront(myDummyHead, 300);

    printList(myDummyHead); // Вывод: 300 -> 200 -> 100 -> NULL

    deleteValue(myDummyHead, 200);

    printList(myDummyHead); // Вывод: 300 -> 100 -> NULL

    // 3. Уборка динамического мусора (саму myDummyHead уберет компилятор C++)
    clearList(myDummyHead);

    return 0; // При выходе из main myDummyHead растворяется
}
```

## Резюме характеристик

| Характеристика               | Описание                                                                                                                                                            |
| :--------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Локация фиктивной головы** | Стек (Stack). Объявляется как локальная/глобальная переменная (обычный объект структуры `Node`).                                                                    |
| **Локация полезных данных**  | Куча (Heap). Реальные узлы создаются динамически через системный вызов `new`.                                                                                       |
| **Синтаксис обращения**      | Сама голова: через точку `(dummyHead.next)`. Динамические узлы: через стрелку `(current->next)`.                                                                    |
| **Очистка памяти (Delete)**  | В цикле очистки запрещено использовать `delete` на саму фиктивную голову. Только на узлы, идущие после неё.                                                         |
| **Преимущество**             | Полное отсутствие "особых случаев" (if пусто), плюс нет риска забыть очистить сам фиктивный узел в конце работы программы (автоматическое управление ОЗУ на стеке). |

## Контрольные вопросы для самопроверки

- Объясните с точки зрения системной архитектуры оперативной памяти, чем "Статическая фиктивная голова" списка отличается от "Динамической"? В чем заключается превосходство в производительности при инициализации статического узла?
- Почему при обращении к фиктивному узлу в теле функции `insertFront` для доступа к следующему вагону используется оператор "точка" (`dummyHead.next`), а при прохождении по остальным вагонам бегунком мы обязаны использовать стрелку (`current->next`)?
- В функции удаления `deleteValue` бегунок `current` инициализируется строкой `Node* current = &dummyHead;`. Какую критическую техническую функцию выполняет символ амперсанда `&` перед именем статической головы? Что произойдет без него?
- Почему в функции очистки `clearList` разработчик категорически не имеет права вписать инструкцию `delete &dummyHead;` вместе со всеми остальными динамическими узлами? Какова будет реакция операционной системы на попытку такого удаления?
- Обоснуйте, почему параметр в функции `insertFront` передается по ссылке `Node& dummyHead`, а не просто по значению `Node dummyHead`. Что случилось бы с добавленными данными (динамическими узлами), если бы мы совершили такую синтаксическую ошибку и передали объект головы по значению (копированием)?
