# Перегрузка функций. Пример использования. Протокол прохождения контрольного примера

## Теоретическая база: Полиморфизм времени компиляции (Ad-hoc Polymorphism)

В классическом языке C каждая функция должна была иметь уникальное имя на всю программу. Если требовалось написать функцию поиска модуля числа для целых чисел, вещественных чисел и длинных целых чисел, приходилось придумывать три разных имени (например, `abs()`, `fabs()`, `labs()`). Это перегружало программиста когнитивной нагрузкой: ему приходилось запоминать десятки имен, хотя логическое _действие_ было одним и тем же.

В C++ эта проблема изящно решается с помощью механизма **Перегрузки функций (Function Overloading)**.
Перегрузка позволяет создавать несколько функций с **абсолютно одинаковым именем**, но с разными входными параметрами (разными интерфейсами приема).

Это простейшая форма статического полиморфизма (полиморфизма времени компиляции или ad-hoc polymorphism). Компилятор C++ способен самостоятельно "догадаться", какую именно из одноименных функций нужно вызвать в данный момент, строго проанализировав **список передаваемых аргументов**.

### Правила различения (Сигнатура функции)

Для того чтобы компилятор мог отличить одну перегруженную функцию от другой, у них должна различаться **сигнатура**.
Сигнатура функции в C++ состоит исключительно из двух вещей:

1.  **Имя функции.**
2.  **Список её формальных параметров** (Количество, Типы данных, Порядок следования).

> **Важно!** Тип возвращаемого значения в сигнатуру **не входит**. Нельзя перегрузить две функции, если они отличаются только типом возвращаемого значения (например, одна возвращает `int`, а другая `double` при одинаковых параметрах компилятор выдаст ошибку неоднозначности).

## Пример использования (Снятие когнитивной нагрузки)

Рассмотрим классическую инженерную задачу: создание математического модуля для вычисления площади (Area) различных геометрических фигур (квадрата, прямоугольника, круга). Вместо создания функций `calcSquareArea`, `calcRectangleArea` и `calcCircleArea` мы создадим единый интерфейс вызова `calcArea`.

```cpp
#include <iostream>

// --- ПРОТОТИПЫ (Overload Set) ---

// 1. Вычисление площади квадрата (принимает 1 целое число)
int calcArea(int side);

// 2. Вычисление площади прямоугольника (принимает 2 целых числа)
int calcArea(int length, int width);

// 3. Вычисление площади круга (принимает 1 число с плавающей точкой)
// Несмотря на то, что параметров 1 (как и у квадрата), ТИП параметра отличается!
double calcArea(double radius);

int main() {
    std::cout << "--- Демонстрация Полиморфизма ---\n";

    // ВЫЗОВ 1: Передаем два int. Компилятор выберет реализацию №2 (прямоугольник)
    std::cout << "Площадь (5, 4):  " << calcArea(5, 4) << "\n";

    // ВЫЗОВ 2: Передаем ровно один double. Компилятор выберет №3 (круг)
    std::cout << "Площадь (2.5):   " << calcArea(2.5) << "\n";

    // ВЫЗОВ 3: Передаем ровно один int. Компилятор выберет №1 (квадрат)
    std::cout << "Площадь (10):    " << calcArea(10) << "\n";

    return 0;
}

// --- ОПРЕДЕЛЕНИЯ (Реализация) ---

int calcArea(int side) {
    std::cout << "[func  ] Сработала логика КВАДРАТА. ";
    return side * side;
}

int calcArea(int length, int width) {
    std::cout << "[func  ] Сработала логика ПРЯМОУГОЛЬНИКА. ";
    return length * width;
}

double calcArea(double radius) {
    std::cout << "[func  ] Сработала логика КРУГА. ";
    const double PI = 3.14159;
    return PI * radius * radius;
}
```

## Протокол прохождения контрольного примера (Name Mangling и Разрешение перегрузки)

Как компилятор понимает, куда направлять поток исполнения, если имена у всех функций одинаковые? Разве линкер (сборщик) не выдаст ошибку дублирования символов?
Секрет кроется в алгоритме компилятора, называемом **Разрешение перегрузки (Overload Resolution)** и механизме изменения имен **Name Mangling**.

Проследим протокол вызова `calcArea(2.5);`.

- **Шаг 1. Фаза Компиляции (Искажение имен / Name Mangling)**
  - Язык C++ не сохраняет имена функций в бинарном файле в их текстовом виде (как `calcArea`).
  - Компилятор склеивает "человеческое" имя и типы параметров, генерируя уникальное, техническое "искаженное" имя (Mangled Name) для среды выполнения.
  - Например (схематично):
    - `calcArea(int)` компилируется в `_Z8calcAreai` (i - значит integer).
    - `calcArea(int, int)` компилируется в `_Z8calcAreaii`.
    - `calcArea(double)` компилируется в `_Z8calcAread` (d - значит double).
  - _Таким образом, для аппаратного обеспечения эти функции абсолютно разные и имеют разные адреса памяти. Иллюзия одинаковых имен существует только для программиста._

- **Шаг 2. Разрешение перегрузки (Overload Resolution)**
  - Компилятор видит строчку `calcArea(2.5);` в `main`.
  - Он анализирует аргументы: один аргумент, литерал `2.5`. По правилам C++ литералы с точкой по умолчанию имеют тип `double`.
  - Компилятор берет множество всех перегруженных функций (Overload Set) и ищет лучшее совпадение (Best Match).
  - Он видит `calcArea(double)` и понимает, что это точное совпадение (Exact Match).
  - Компилятор жестко "зашивает" вместо имени `calcArea` в этой конкретной строке вызов `_Z8calcAread`.

- **Шаг 3. Исполнение (Runtime)**
  - Создается стандартный стековый фрейм для функции `_Z8calcAread`. Значение `2.5` помещается в регистры сопроцессора с плавающей точкой (FPU/SSE).
  - Выполняется умножение площади круга. Возвращается результат. Конец фрейма.

## Проблема автоматического приведения типов (Type Promotion & Coercion)

Алгоритм Overload Resolution является сложным экспертным механизмом. Сбои возникают, когда "точного совпадения" (Exact Match) нет, и компилятор начинает легально сужать или расширять типы данных (Type Casting), чтобы подогнать аргумент под одну из имеющихся функций.

**Ситуация неоднозначности (Ambiguity):**
Представим, что у нас есть:
`void printValue(int x);`
`void printValue(double x);`

Мы делаем вызов:
`printValue(5.5f);` // Буква 'f' означает тип float (32-бит).

1. Точного совпадения `float` нет.
2. Компилятор пытается найти путь конвертации.
3. `float` можно благополучно преобразовать (расширить) в `double`.
4. `float` можно усечь (собрать дробную часть) до `int`.
5. По стандарту C++ расширение до `double` имеет приоритет перед усечением в целое `int`.
6. Компилятор вызовет функцию `printValue(double)`.

**НО! Ошибка компиляции `Call to overloaded function is ambiguous`:**
Представим:
`void printData(int x);`
`void printData(char x);`

Вызываем:
`printData(3.14);` // double

Компилятор зависает: перевести `double` в `int` (усечение) и перевести `double` в `char` (еще более жесткое усечение) — это преобразования _одной степени тяжести_. Компилятор не будет брать на себя ответственность решать, чьими данными пожертвовать, и прервет компиляцию с сообщением об "ambiguous call" (неоднозначном вызове).

## Резюме характеристик

| Характеристика            | Оценка / Описание                                                                                                         |
| :------------------------ | :------------------------------------------------------------------------------------------------------------------------ |
| **Синонимы**              | Статический полиморфизм, Ad-hoc polymorphism                                                                              |
| **Основа различия**       | Сигнатура параметров (Типы, Количество, Порядок)                                                                          |
| **Участие типа возврата** | **НЕТ**. Компилятор не учитывает тип возвращаемого значения при разрешении конфликта                                      |
| **Механизм трансляции**   | _Name Mangling_ - кодирование типов параметров в машинное имя функции                                                     |
| **Время работы**          | Решается на этапе компиляции (Compile-Time). Накладные расходы (Overhead) в Runtime - `O(1)` (нулевые накладные расходы). |
