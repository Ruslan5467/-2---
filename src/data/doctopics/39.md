# Работа с файлами. Способы доступа к данным. Прямой доступ. Последовательный доступ.

## Теоретическая база: Стратегии перемещения по файлу

Когда программа работает с оперативной памятью (с массивом данных), она имеет так называемый Произвольный (Прямой) доступ: чтобы получить 500-й элемент массива `arr[500]`, процессору не нужно перебирать предыдущие 499. Он просто вычислит адрес сдвига и заберет ячейку памяти за время $O(1)$.

Однако файлы хранятся на внешних накопителях (дисках), и природа работы с ними абстрагирована операционной системой в концепцию **Потока (Stream)** или "ленточной последовательности байтов".

В зависимости от физической структуры файла и требований алгоритма, операционная система предоставляет программисту два кардинально различающихся архитектурных подхода (режима) для чтения и записи файлов:

1.  **Последовательный доступ (Sequential Access)**
2.  **Прямой / Произвольный доступ (Random / Direct Access)**

---

## 1. Последовательный доступ к файлу (Sequential Access)

Исторически этот режим появился еще во времена накопителей на магнитной ленте (бобинах), когда чтобы прочитать 10-ю песню, нужно было физически перемотать ленту через первые 9.

В современных реалиях программирования **Последовательный доступ** означает, что программа читает (или пишет) файл строго от начала к концу (байт за байтом, символ за символом, строка за строкой).

### Механика работы и указатель (File Pointer):

Когда вы открываете файл операционной системой, она создает системную невидимую "закладку" (Указатель позиции файла / File Position Cursor), установленную на самый 0-й байт (начало файла).

- Когда вы выполняете функцию чтения (например, "прочитай 12 байт"), курсор автоматически сдвигается на 12 байт вправо.
- Следующий вызов функции чтения начнет чтение именно с 13-го байта.
- Движение идет строго в одном направлении. По умолчанию нельзя сказать: "а ну-ка прыгни назад".

### Где это является стандартом индустрии?

- **Текстовые файлы (`.txt`, `.csv`, `.log`, `.json`):** Это самая важная сфера применения! В текстовом файле строки имеют переменный размер: слово "Hi" (2 байт), а слово "Hello" (5 байт). Вы математически _не можете_ угадать, на каком конкретно байте начинается 100-я строчка в файле. Вы обязаны последовательно прочитать весь файл с начала, считая символы переноса строки `\n` как маркеры, пока не доберетесь до нужного места.
- **Чтение потокового мультимедиа:** Видео или аудио из интернета транслируется потоком.
- **Чтение из терминала или пайпов:** Работа с `std::cin` (стандартный ввод) — это чистый последовательный поток. Вы не можете "отмотать" клавиатурный ввод пользователя назад.

_Временная асимптотика:_ Для чтения элемента в середине файла процессору потребуется $O(N)$ времени (нужно пропустить всё предшествующее).

## 2. Прямой (Произвольный) доступ к файлу (Direct/Random Access)

Прямой доступ — это способность программы скомандовать операционной системе: _"Смести курсор (указатель) файла мгновенно на байт номер `8405` и прочитай порцию данных оттуда!"_ — игнорируя чтение всех предшествующих байтов или строк.

### Механика позиционирования:

В языках программирования (таких как C++) для этого существуют специальные функции ручного позиционирования (например, `seekg()` — seek get для чтения, и `seekp()` — seek put для записи).

Функция `seek` принимает два параметра:

1. **Смещение (Offset):** Количество байтов.
2. **Опорная система координат (Anchor):** Откуда отсчитывать (От начала файла `beg`, от текущей позиции `cur`, или с конца файла `end`, отматывая в минус).

Пример концепции:

```cpp
// Прыгнуть ровно на 1024-й байт от самого начала файла
file.seekg(1024, std::ios::beg);
// Прочитать блок данных
file.read(buffer, 256);
```

### Где это является стандартом индустрии?

Прямой доступ используется в **Бинарных файлах (Binary файлов)** или специализированных бинарных структурах (БД), где размер единицы данных строго фиксирован.

- **Базы данных (СУБД / SQL):** Если в файле базы данных каждая запись (Struct) "Студент" весит ровно 100 байт (например, 20 байт имя + 80 байт предметы), то если вам нужна карточка 5-го студента, программе не нужно читать файл с начала! Она мгновенно вычисляет смещение: `(5 - 1) * 100 байт = 400`. Программа делает прямой прыжок на 400-й байт $O(1)$ и загружает 100 байт студента.
- **Медиа-движки и Архиваторы (.zip, .mp4):** Чтобы показать видео с 15-й минуты, плеер должен иметь возможность прыгнуть прямо в нужный чанк бинарного файла, а не декодировать первые 14 минут.
- **Обновление файла по месту:** Прямой доступ (`seekp`) позволяет вам переписать маленькое слово в середине огромного гигабайтного файла, без необходимости перезаписывать весь файл целиком (что потребовалось бы при текстовом последовательном доступе).

## Резюме характеристик

| Характеристика                   | Последовательный доступ (Sequential)                                      | Прямой доступ (Direct/Random)                                                                        |
| :------------------------------- | :------------------------------------------------------------------------ | :--------------------------------------------------------------------------------------------------- |
| **Механика курсора**             | Автоматически двигается только вперед, как бобина ленточного магнитофона. | Прыгает в любую физическую координату по команде "Seek" (Позиционирование).                          |
| **Типичный формат данных**       | Текстовые файлы (`.txt`, `.log`, `.xml`), Терминалы.                      | Бинарные блоки (Структуры `Struct`, Базы Даных, Архивные блоки).                                     |
| **Размер ячеек / строк**         | Переменный (Variable length). Строки разной длины разделены метками `\n`. | Строго фиксированный размер (Fixed length). Иначе рассчитать математический адрес прыжка невозможно. |
| **Скорость поиска (в середину)** | Низкая $O(N)$ (надо физически "прожевать" всё с начала до нужного места). | Высокая $O(1)$ (мгновенный прыжок в вычисленный байт).                                               |
| **Ресурсная эффективность**      | Очень экономно пожирает ОЗУ, читает порциями.                             | Требует математически точного контроля над структурами данных (С/С++ Layouts).                       |

## Контрольные вопросы для самопроверки

- Объясните главную системную причину, по которой программисты концептуально лишены возможности использовать алгоритмы "Прямого доступа", если они работают с классическим текстовым файлом (.txt) и им срочно понадобилось прочитать 185-ю текстовую строчку?
- Как функция библиотеки позиционирования (например, аналог `seek`) математически рассчитывает "сдвиг" в бинарном файле базы данных, если нам нужно получить адрес (offset) записи номер `42`, при условии, что размер байтовой структуры каждой записи (Struct) строго равен $64$ байтам?
- Объясните смысл и назначение невидимого "Системного курсора (Указателя файла)". Что происходит с этим курсором при выполнении стандартной команды чтения `read(10 байт)`?
- Предварительно скомпилированная "Структура C++ (Struct)", сохраненная в бинарный файл, идеально подходит для Прямого доступа. Но если в структуре есть строковое поле `std::string name` (которое динамически расширяет свой размер в куче), почему запись такого поля в файл разрушит всю математику прямого доступа? Что (какие типы строк `char`) нужно использовать вместо него?
- Можем ли мы использовать команду "прыжка (seek) назад" (например, отмотать на начало `0`) в последовательном текстовом файле, который мы в данный момент _читаем_ на диске, если вдруг поняли, что пропустили нужную строку? И можем ли мы применить такую же перемотку для потока терминального ввода пользователя `std::cin`?
