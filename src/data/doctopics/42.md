# Сложность алгоритмов. Оценка временной сложности

## Теоретическая база: Зачем нужна оценка сложности?

В программировании для решения одной и той же задачи почти всегда существует несколько разных алгоритмов. Например, отсортировать массив можно "Пузырьком", а можно с помощью "Быстрой сортировки" (QuickSort). Возникает фундаментальный вопрос: **Какой алгоритм лучше?**

Интуитивный подход — просто запустить алгоритм и замерить время работы в секундах (с помощью секундомера). Однако этот подход в Computer Science считается неверным, так как время в секундах зависит от множества побочных факторов:

1. Мощности процессора (современный процессор выполнит плохой алгоритм быстрее, чем процессор из 1995 года хороший).
2. Загруженности операционной системы (фоновые процессы, антивирусы).
3. Используемого языка программирования (C++ отработает в разы быстрее, чем Python на одинаковом алгоритме).

Чтобы сравнивать _математическую эффективность самих алгоритмов_, независимо от "железа" и языка, ученые ввели понятие **Сложности (Complexity)**.

Оценка сложности изучает асимптотику: **как изменяется количество необходимых операций процессора при увеличении объема входящих данных ($N$)**.

---

## Что такое Временная сложность (Time Complexity)?

**Временная сложность** — это функция от длины входных данных (обычно обозначается как $N$), которая отражает количество _элементарных операций_, выполняемых алгоритмом.

**Элементарные операции ($O(1)$)** — это базовые шаги процессора, время выполнения которых не зависит от размера данных $N$. К ним относятся:

- Математические операции (`+`, `-`, `*`, `/`).
- Сравнения (`>`, `<`, `==`).
- Присваивание значений (`x = 5`).
- Доступ к элементу массива по индексу (`arr[5]`).

Если для массива из 10 элементов алгоритм делает 10 операций, а для массива из 1000 элементов — 1000 операций, говорят, что время работы алгоритма растет _линейно_ (прямо пропорционально размеру данных).

---

## Нотация «О» большое (Big O Notation)

В индустрии программирования универсальным стандартом для записи сложности стала асимптотическая нотация **"О" большое (Big O)**.
Она описывает **Худший случай (Worst-case scenario)** работы алгоритма. Это верхняя граница: программист должен знать максимальное количество времени (или памяти), которое алгоритм может потребовать при неудачном стечении данных.

### Главные математические правила "Big O":

1. **Отбрасывание констант:** В теории сложности нас интересует порядок роста (график функции на бесконечности), а не точные числа. Если ваш алгоритм делает $5N$ операций (например, 5 действий внутри линейного цикла), константу $5$ отбрасывают.
   - **Математика:** $O(5N) \rightarrow O(N)$.
2. **Отбрасывание меньших терминов (доминанта):** При больших $N$ алгоритм определяется своим самым "медленным" (тяжелым) компонентом. Если алгоритм состоит из двух независимых частей (например, сначала цикл $N^2$, а затем цикл $N$), более быстрая часть $N$ на фоне миллиардов итераций $N^2$ не играет никакой роли.
   - **Математика:** $O(N^2 + 5N + 1000) \rightarrow O(N^2)$.

## Основные классы временной сложности

Ранжирование алгоритмов от самых быстрых (лучших) к самым медленным:

### 1. $O(1)$ — Константное время

- **Смысл:** Идеально. Время работы алгоритма вообще не зависит от размера входных данных! Для массива из 10 элементов и для массива из 1 миллиона элементов программа потратит одно и то же количество тактов.
- **Пример в коде:** Обращение к элементу массива по индексу `int a = arr[5];`, проверка четности числа.
- **Структуры данных:** Hash Tables (Хэш-таблицы).

### 2. $O(\log N)$ — Логарифмическое время

- **Смысл:** Превосходно. При каждом следующем шаге алгоритма объем данных, с которым нужно работать, уменьшается в два раза! График функции растет крайне медленно. Для 1 Секстиллиона элементов потребуется всего около 64 операций!
- **Пример в коде:** Бинарный поиск (Binary Search) в отсортированном массиве или поиск в Бинарном дереве (BST).

### 3. $O(N)$ — Линейное время

- **Смысл:** Нормально. Время выполнения растет ровно пропорционально росту входных данных $N$. Увеличили массив в 10 раз — алгоритм стал работать в 10 раз дольше.
- **Пример в коде:** Один цикл `for`, обходящий весь массив (например, поиск максимального элемента или линейный поиск суммы чисел).

### 4. $O(N \log N)$ — Линейно-логарифмическое время

- **Смысл:** Хорошо. Это предел эффективности для алгоритмов, которые основаны на сравнении элементов.
- **Пример в коде:** Лучшие алгоритмы сортировки (Быстрая сортировка QuickSort, Сортировка слиянием MergeSort).

### 5. $O(N^2)$ — Квадратичное время

- **Смысл:** Плохо (для больших данных). При увеличении входных данных в 10 раз, время работы увеличивается в $10^2 = 100$ раз! Такие алгоритмы непригодны для баз данных с миллионами записей (они зависнут на часы или дни).
- **Пример в коде:** Два вложенных цикла `for` (один внутри другого). Например, "Сортировка Пузырьком" (Bubble Sort) или обход двумерной матрицы $N \times N$.

### 6. $O(2^N)$ — Экспоненциальное время (и выше)

- **Смысл:** Катастрофа. Алгоритм удваивает время работы при добавлении всего ОДНОГО нового элемента во входные данные $N$.
- **Пример в коде:** Рекурсивный расчет чисел Фибоначчи без оптимизации (Мемоизации), брутфорс (перебор) всех возможных комбинаций пароля.

## Резюме характеристик

| Класс сложности $O(...)$ | Имя                     | Качество для больших $N$ | Пример алгоритма из индустрии                                |
| :----------------------- | :---------------------- | :----------------------- | :----------------------------------------------------------- |
| **$O(1)$**               | Константное             | 🌟 Идеал                 | Извлечение элемента из массива `M[5]`.                       |
| **$O(\log N)$**          | Логарифмическое         | 🟢 Прекрасно             | Бинарный поиск по отсортированной базе.                      |
| **$O(N)$**               | Линейное                | 🟡 Нормально             | Поиск слова в неструктурированном `.txt` тексте.             |
| **$O(N \log N)$**        | Линейно-логарифмическое | 🟡 Хорошо                | Эффективные Сортировки базы данных (Merge Sort).             |
| **$O(N^2)$**             | Квадратичное            | 🔴 Плохо                 | Вложенные циклы, наивная сортировка пузырьком.               |
| **$O(2^N)$**             | Экспоненциальное        | 💀 Катастрофа            | Глупый перебор криптографических ключей, рекурсия Фибоначчи. |
